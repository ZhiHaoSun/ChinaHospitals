# MedTour AI 跨境医疗旅游智能规划系统 PRD

版本：v1.1  
日期：2026-07-12  
来源文档：`product.md`  
产品形态：面向海外用户的 AI 医疗旅游规划 Web / Mobile Web 应用  
目标市场：新加坡、美国、马来西亚等海外用户来华进行医疗消费与短途旅游  

## 1. 产品概述

MedTour AI 是一套基于多智能体架构的跨境医疗旅游智能规划系统，面向希望来中国完成近视矫正、牙科种植、高端体检、医美等项目的海外中产及高净值用户。

系统通过用户需求采集、医疗机构匹配、就医周期锁定、差旅规划、费用测算和时间表生成，为用户输出一份可执行的医疗旅游方案。方案应同时回答三个核心问题：

- 我适合去中国哪个城市、哪家医院或机构做这个项目？
- 整个行程需要几天、每天怎么安排？
- 相比在本国完成同等项目，我是否真的节省成本？

当前版本已经具备可演示的端到端原型：Vercel 部署 Web 界面、本地 FastAPI 后端、本地确定性 Planner、多 Agent 规划链路、Pydantic Schema 校验、城市对比、计划详情页、保险与账单提示、医院国际部联系路径、费用拆分、出行准备清单和 PDF 导出。

## 2. 产品目标

### 2.1 用户目标

- 快速判断来华医疗消费是否值得。
- 获得清晰可信的医疗项目周期、费用和风险提示。
- 获得可直接执行的航班、酒店、交通和就医时间表。
- 降低跨境预约、语言、支付、证件、交通和复查安排的不确定性。

### 2.2 业务目标

- 将海外用户的模糊医疗旅游意向转化为结构化商机。
- 建立医疗机构、国际部、高端私立机构和差旅服务的推荐入口。
- 通过 AI 规划报告提升用户咨询转化率。
- 为后续人工顾问、预约代办和增值服务提供标准化用户档案。
- 将复杂跨境医疗行程拆解为可报价、可确认、可交付的服务包。
- 为医院国际部、医疗顾问、保险服务和差旅供应商提供高意向、结构化、可追踪的用户需求。

### 2.3 产品成功指标

- 首份完整规划报告生成成功率 ≥ 95%。
- 完整报告生成耗时 P95 ≤ 15 秒。
- 用户完成需求采集表单转化率 ≥ 60%。
- 用户点击“预约顾问 / 获取报价 / 保存方案”转化率 ≥ 20%。
- 报告中关键字段缺失率 ≤ 3%。
- 医疗费用与差旅费用均展示来源、更新时间或置信度。
- 用户从报告页进入顾问转接的线索完整率 ≥ 80%。
- 每个城市方案的关键风险、保险确认项和医院联系路径覆盖率 ≥ 90%。
- AI 生成报告通过 Schema 校验后再进入前端展示，Schema 校验失败必须返回可诊断错误。

## 2.4 商业价值

MedTour AI 的商业价值来自“高决策成本 + 高客单价 + 高信任门槛”的交叉场景。

对用户而言，跨境医疗不是简单旅游消费，而是包含医疗安全、费用透明、医院可信度、保险报销、签证入境、支付工具、交通住宿和术后恢复的复合决策。用户通常不知道从哪里开始，且很难判断中国方案是否真的比本国方案更便宜、更快或更适合。

对新加坡用户而言，私立专科服务、牙科种植、近视矫正、高端体检和肿瘤二次意见可能价格较高；公立体系虽然质量稳定，但非紧急专科预约可能存在排队和转诊流程。中国方案有价格、语言和航班便利性优势，但用户需要完整总成本和官方医院路径。

对美国用户而言，医疗价格高度不透明。即使有保险，deductible、out-of-network billing、pre-authorization、claim denial 和 surprise bill 都会造成决策压力。中国方案只有在纳入机票、酒店、翻译、复查、保险报销和并发症预案后，才具备真实比较意义。

对医院 / 顾问 / 机构而言，MedTour AI 可以将低质量的“随便问问”变成高质量线索：用户目的、预算、时间窗、出发地、保险情况、风险项、偏好和待确认事项都已经结构化，顾问首次沟通效率更高，转化路径更短。

## 2.6 市场切入与 Go-to-Market

首批市场建议聚焦：

- 新加坡：中文 / 英文用户多，赴华航班便利，对眼科、齿科、体检和二次意见有明确需求。
- 美国华人及国际用户：医疗价格高且保险复杂，适合用“总成本透明 + 中国医院路径确认”作为切入点。
- 马来西亚：中文沟通和区域航班优势明显，可作为第二阶段扩展市场。

首批项目建议聚焦：

- 眼科近视矫正：周期短、价格可比较、行程硬约束清晰。
- 牙科种植 / 修复：本国价格高、用户对跨境方案敏感。
- 高端体检 / 肿瘤筛查：适合套餐化和二次意见。
- 肿瘤或血液病二次意见：高客单价但需要更强合规与人工顾问介入。

获客入口：

- SEO 内容：Singapore to China eye surgery cost、China dental implant medical travel、China hospital international department 等。
- 社群与内容平台：面向海外华人、新加坡家庭、美国高医疗账单用户的解释型内容。
- 医疗顾问合作：将 AI 报告作为顾问 intake 工具。
- 医院国际部合作：为海外咨询用户提供预筛选和行程说明。

### 3.2 核心使用场景

- 用户已有明确项目，例如“我想去上海做 SMILE Pro 全飞秒”。
- 用户只有模糊需求，例如“我想体检，顺便在广州玩几天”。
- 用户正在比较本国医疗价格和中国医疗价格。
- 用户需要知道最短请假天数和最佳出发日期。
- 用户需要向家人或同行人解释预算和行程安排。

## 4. 范围定义

### 4.1 MVP 范围

- 多语言引导式输入：中文、英文、马来文界面下，通过选择题、短问题、搜索选择和日期控件完成输入。
- 极简输入模式：首屏不使用开放式长文本输入，支持用户通过“医疗目的 + 国籍 / 居住地 + 出发城市 + 大致时间 / 停留时长”生成初版方案，其他信息通过智能默认值和后续确认补齐。
- 用户画像结构化：国籍、居住地、出发城市、证件类型、预算等级、医疗项目、期望日期、同行人数。
- 医疗方案推荐：城市、医院 / 机构、科室、项目周期、费用区间、注意事项。
- 多城市方案生成：默认生成最多 4 个中国城市方案，每个方案包含医疗建议、详细时间表、费用估算、出行准备和风险提示。
- 差旅方案推荐：航班、酒店、本地交通、轻度旅游安排。
- 全额费用评估：医疗、机票、酒店、餐饮交通、签证、复查预估、本国对比价格、净节省。
- 小时级时间表：按天、按小时展示医疗节点和旅行节点。
- 可编辑时间表：用户可修改出发日期、停留天数、酒店档位、旅游强度、航班偏好后重新生成时间表。
- 出行准备助手：签证 / 免签判断、签证申请步骤、支付宝国际用户设置、支付备用方案、出发前检查清单。
- 报告导出：在线查看、复制链接、下载 PDF。
- 人工顾问转接：将结构化需求和 AI 报告传给客服 / 顾问。

### 4.2 非 MVP 范围

- 直接完成医院官方挂号支付。
- 直接售卖机票和酒店。
- 作为医疗诊断或治疗建议替代医生意见。
- 存储完整护照扫描件或高度敏感医疗影像。
- 保证医院实际可预约号源。

## 5. 产品原则

- 医疗安全优先：不输出诊断结论，不承诺治疗效果。
- 可解释优先：所有费用、时间和推荐必须展示依据或置信度。
- 动态验证优先：签证政策、航班价格、酒店可入住资质、医院流程不可硬编码。
- 行程可执行优先：时间表必须留出合理交通、等待、休息和术后恢复时间。
- 跨语言友好：海外用户应能在中文、英文、马来文界面下通过同一套引导式选择完成需求输入。
- 低摩擦优先：首轮输入只收集生成初版方案所必需的信息，其余字段通过默认值、选择卡片和 Agent 追问逐步补齐。
- 用户确认优先：当 Agent 对关键判断不确定或存在多个合理选择时，系统必须让用户确认后再锁定方案。

## 7. 用户体验流程

### 7.1 首次生成规划流程

1. 用户进入首页。
2. 系统展示引导式问题：“What medical care are you planning for?”
3. 用户通过卡片选择医疗目的：
   - Eye surgery
   - Dental care
   - Health checkup
   - Medical aesthetics
   - Other common procedures
4. 系统根据医疗目的展示下一步短问题。例如眼科路径继续询问：
   - 期望项目：SMILE Pro / LASIK / ICL / Not sure
   - 是否已有度数范围：Yes / No / Not sure
5. 系统继续询问旅行基础信息：
   - 国籍 / 居住地
   - 出发城市
   - 计划出发时间：具体日期 / 月份 / flexible
   - 可接受停留时长：3-4 天 / 5-7 天 / 8 天以上 / 不确定
   - 是否接受冬季或淡季出行：Yes / No / Depends on price
6. User Profiler Agent 根据选择结果结构化用户画像，并标记默认值、已确认字段和待确认字段。
7. 对非必填字段，系统使用可见默认值并允许用户稍后修改：
   - 预算等级默认“平衡”
   - 同行人数默认“1 人”
   - 酒店偏好默认“医院附近、可外宾入住、评分较高”
   - 旅游强度默认“轻度”
   - 是否持有中国内地身份证默认“否 / 未确认”
8. 若 Agent 识别结果低置信或有多个选择，系统以选择卡片追问，例如“你想优先上海、广州，还是让系统按性价比推荐？”
9. 用户点击“生成医疗旅游方案”。
10. 前端进入生成状态，展示多 Agent 进度：
   - 正在确认医疗项目周期
   - 正在匹配医院和城市
   - 正在检索航班与酒店
   - 正在计算总预算
   - 正在生成小时级行程
11. 系统在 15 秒内返回完整报告。
12. 用户查看摘要卡片：
   - 推荐城市
   - 推荐医院 / 机构
   - 最短行程天数
   - 总预算
   - 预计节省金额
   - 风险提示
13. 用户继续查看完整报告、调整条件、修改时间表、导出 PDF 或联系顾问。

### 7.2 医疗项目选择流程

1. 用户进入项目选择区。
2. 系统展示项目类别：
   - 眼科：全飞秒、半飞秒、ICL、白内障等
   - 牙科：种植牙、正畸、贴面等
   - 体检：高端体检、肿瘤筛查、心脑血管筛查等
   - 医美：皮肤、注射、抗衰、外科医美等
3. 用户选择项目后，系统展示必要信息输入项。例如近视手术需要：
   - 近视度数范围
   - 散光情况
   - 是否佩戴隐形眼镜
   - 是否有既往眼病
   - 期望术式
4. 系统提示“该信息仅用于行程与预算预估，不构成诊断”。
5. 用户提交后，Medical Consultant Agent 输出适配城市、医院和就医周期。
6. 如项目存在强医学前置条件，系统显示“不确定性提示”，例如“是否可做全飞秒需以术前检查为准”。

### 7.3 多城市目的地与医院推荐流程

1. Medical Consultant Agent 基于用户项目、预算、语言需求和目标日期检索知识库。
2. 系统默认生成最多 4 个中国城市方案；当可行城市不足 4 个时，只展示满足医疗与行程约束的方案。
3. 城市候选应覆盖不同策略：
   - 高端效率型：国际部 / 特需 / 私立机构
   - 性价比型：公立三甲国际部或特需门诊
   - 旅游友好型：医疗资源与城市游玩兼顾
   - 航班便利型：从用户出发城市直飞或中转更少
4. 每个城市方案必须包含：
   - 城市
   - 医院 / 机构名称
   - 推荐原因
   - 预计医疗费用
   - 差旅费用估算
   - 总费用估算
   - 预计净节省
   - 必要天数
   - 该城市的小时级时间表
   - 语言服务
   - 预约难度
   - 签证 / 入境与支付准备影响
   - 主要风险和不确定项
   - 数据来源与更新时间
5. 系统应以横向对比卡片展示多个城市方案，并标记推荐排序。
6. 用户可以选择一个方案作为主方案，也可以进入对比视图查看详细时间表和费用拆分。
7. 用户选定主方案后，Travel Coordinator Agent 和 Timeline Generator 对该方案进行精细化重算。
8. 未选中的城市方案保留为备选，用户可随时切换并重新生成。

### 7.4 差旅规划流程

1. 系统读取医疗硬性时间窗，例如：
   - Day 1 术前检查
   - Day 2 手术
   - Day 3 术后复查
2. Travel Coordinator Agent 根据出发城市和每个候选医疗城市反向推导航班。
3. 系统优先推荐满足以下条件的航班：
   - 抵达时间早于术前检查至少 4-6 小时
   - 回程时间晚于复查结束至少 4-6 小时
   - 避免红眼航班影响术前状态
4. 系统推荐医院周边酒店：
   - 距离医院步行 15 分钟以内或车程 20 分钟以内
   - 明确支持外宾入住
   - 评分、价格和取消政策可见
5. 系统安排轻度旅游活动：
   - 不与医疗节点冲突
   - 近视手术后避免强光、剧烈活动和长时间用眼
   - 牙科或医美后避免不适宜饮食和高强度行程
6. 用户可以调整：
   - 多住几晚
   - 更换酒店档位
   - 增加旅游天数
   - 降低预算
   - 更换城市
7. 系统重新计算所选城市方案的费用和时间表；若用户切换城市，则重新计算该城市方案。

### 7.5 费用评估流程

1. Finance Estimator Agent 汇总医疗、差旅和本国对比数据。
2. 系统展示费用总览：
   - 中国医疗费用
   - 往返机票
   - 酒店
   - 本地交通
   - 餐饮与杂费
   - 签证或入境相关费用
   - 回国复查费用
   - 预留缓冲费用
3. 系统按用户本币展示总价，并保留人民币明细。
4. 系统展示本国同等项目参考价。
5. 系统计算：
   - 总花费
   - 本国参考价
   - 预计净节省
   - 节省比例
   - 价格不确定区间
6. 若净节省为负，系统不强行包装为划算，应展示“该方案以服务体验 / 旅行价值为主，成本优势不明显”。
7. 用户可以切换经济、高端、豪华预算档位查看差异。

### 7.6 小时级时间表流程

1. Timeline Generator 根据医疗硬约束生成基础时间轴。
2. 系统以天为单位展示：
   - 航班
   - 入境和交通
   - 医院签到
   - 检查
   - 手术 / 项目
   - 复查
   - 用餐
   - 休息
   - 轻度旅游
   - 回程
3. 每个节点包含：
   - 开始时间
   - 结束时间
   - 地点名称和完整地址
   - 交通方式
   - 预计费用和币种
   - 费用拆分
   - 预订状态
   - 数据来源和更新时间
   - 注意事项
4. 不同类型节点必须展示对应详细字段：
   - 航班：航司、航班号、起降机场、航站楼、起飞 / 到达时间、飞行时长、中转信息、行李规则、票价、税费、退改规则、价格更新时间。
   - 酒店：酒店名称、星级 / 评分、完整地址、距离医院、步行 / 车程时间、房型、入住 / 退房时间、每晚价格、税费、总价、取消政策、是否确认支持外宾入住。
   - 本地交通：出发地、目的地、路线、交通方式、预计时长、预计费用、等待时间、是否建议预约接送。
   - 医疗节点：医院 / 机构名称、科室、国际部 / 特需 / 私立服务类型、地址、楼层或报到点、联系人或预约状态、预计费用拆分、所需材料、注意事项。
   - 旅游 / 用餐节点：地点名称、地址、距离、预计停留时间、门票 / 餐饮预算、活动强度、是否适合术后恢复期。
5. 系统自动插入缓冲时间：
   - 机场到医院 / 酒店交通缓冲
   - 医院排队等待缓冲
   - 术后休息缓冲
   - 国际航班提前到达机场缓冲
6. 如果用户调整航班或酒店，时间表自动重算。
7. 用户可以展开每个节点查看详细成本和替代方案。
8. 用户可以导出时间表为 PDF 或日历格式。

### 7.7 顾问转接流程

1. 用户在报告页点击“联系顾问确认预约”。
2. 系统展示授权确认，说明将共享：
   - 用户联系方式
   - 国籍 / 居住地
   - 医疗项目需求
   - 预算和目标日期
   - AI 生成方案
3. 用户填写联系方式和首选沟通语言。
4. 系统创建顾问线索，并将 AI 报告附加到 CRM 或后台。
5. 顾问可在后台查看报告、补充人工建议、标记下一步：
   - 待联系
   - 已联系
   - 待医院确认
   - 已预约
   - 已取消

### 7.8 出行准备助手流程

1. 系统在生成报告后自动创建“出发前准备”模块。
2. Pre-Trip Readiness Agent 根据用户国籍、护照类型、行程天数、入境目的、出发国家和抵达城市判断准备事项。
3. 系统将准备事项分为：
   - 入境与签证
   - 支付与本地 App
   - 医疗文件
   - 通信与网络
   - 保险与紧急联系人
   - 行李与术后护理用品
4. 用户进入“入境与签证”卡片，系统先判断：
   - 是否可能享受免签或过境免签
   - 是否需要提前申请签证
   - 是否需要人工顾问确认
5. 如用户可能免签，系统展示：
   - 可停留天数
   - 适用入境目的
   - 护照有效期要求
   - 建议携带材料，如往返机票、酒店预订、医疗预约确认、行程单
   - 官方政策链接和更新时间
6. 如用户需要签证，系统展示申请步骤：
   - 选择签证类型，通常旅游医疗行程优先提示 L 签或按官方要求确认
   - 检查护照有效期、空白签证页和照片要求
   - 填写中国在线签证申请表或当地签证中心表单
   - 准备行程材料，如机票、酒店、医疗预约或邀请材料
   - 预约或前往签证中心 / 使领馆提交材料
   - 按要求采集指纹或补充材料
   - 支付签证费用并领取护照
   - 上传签证状态，系统更新是否可出票 / 可预约
7. 系统生成倒计时清单：
   - T-60 至 T-45：确认签证 / 免签资格，检查护照
   - T-45 至 T-30：提交签证申请或顾问复核
   - T-21：确认机票、酒店和医院预约材料
   - T-7：安装支付 App、绑定银行卡、准备少量现金
   - T-1：下载离线行程、保存医院地址、确认接机方式
8. 用户进入“支付宝国际用户设置”卡片，系统按手机系统给出下载入口。
9. 系统引导用户完成：
   - 下载 Alipay 官方 App
   - 使用手机号注册或登录
   - 切换英文或用户熟悉语言
   - 按 App 要求完成身份验证
   - 绑定国际银行卡
   - 设置支付密码和安全验证
   - 保存医院、酒店和接送机联系人
10. 系统提示用户准备支付备用方案：
   - 国际信用卡或借记卡
   - 少量人民币现金
   - 可选微信支付设置
   - 医院是否接受境外卡、支付宝、微信支付或现金的确认结果
11. 用户勾选每个准备项后，系统更新“出行准备完成度”。
12. 对未完成的高风险事项，系统在报告页和时间表中显示提醒，例如“签证状态未确认，不建议支付不可退酒店”。

### 7.9 Agent 追问与用户确认流程

1. 任一 Agent 在遇到低置信判断、多个合理选项或高影响决策时，必须向 Orchestrator Agent 发出 confirmation_request。
2. Orchestrator Agent 判断该问题是否阻塞报告生成：
   - 阻塞问题：会影响医疗安全、签证资格、行程可行性、预算大幅变化或不可退订消费。
   - 非阻塞问题：只影响偏好排序、酒店风格、旅游活动等可稍后调整事项。
3. 对阻塞问题，系统暂停锁定最终方案，并向用户展示简短选择卡片。
4. 对非阻塞问题，系统使用默认值继续生成，同时在报告中标记“可调整”。
5. 用户确认后，系统更新用户画像、偏好和 Agent 上下文。
6. 系统仅重新运行受影响的 Agent，避免整份报告从头生成。
7. 所有确认问题必须说明：
   - 为什么需要确认
   - 不同选择的影响
   - 推荐选项
   - 是否可以稍后修改

常见追问示例：

- “你更想优先节省预算，还是优先医院国际部效率？”
- “上海和广州都有合适眼科方案，你想指定城市，还是让系统按总费用推荐？”
- “你的出发日期只写了 8 月。要按周末短假期规划，还是按最低机票价格规划？”
- “该项目可能需要多次复诊。你是否可以接受二次赴华？”
- “签证状态未确认。是否先生成暂定方案，还是联系顾问确认入境要求？”

### 7.10 时间表修改与重新生成流程

1. 用户在报告页进入“编辑时间表”。
2. 系统展示可调整偏好：
   - 出发日期或日期范围
   - 总停留天数
   - 医疗城市
   - 酒店档位和距离
   - 航班偏好：直飞、低价、早到、晚回、避免红眼
   - 旅游强度：无旅游、轻度、半日游、一日游
   - 是否需要接送机、翻译或同行人安排
   - 每日最早出发时间和最晚返回酒店时间
3. 系统将医疗硬约束锁定显示，例如“Day 1 术前检查、Day 2 手术、Day 3 复查不可随意移动”。
4. 用户修改偏好后点击“重新生成时间表”。
5. Timeline Generator 只重算受影响的航班、酒店、交通、旅游和缓冲时间。
6. 若用户修改会破坏医疗硬约束，系统必须提示原因并给出替代方案。
7. 系统生成新版本时间表，并展示变化摘要：
   - 总价变化
   - 总天数变化
   - 医疗节点是否变化
   - 酒店 / 航班变化
   - 风险变化
8. 用户可保留旧版本、接受新版本，或继续调整。
9. 系统保存最近 3 个时间表版本，便于用户对比和回退。

## 8. 功能需求

### FR-001 多语言引导式输入

系统应支持用户在中文、英文、马来文界面下，通过选择题、短问题、搜索选择和日期控件输入医疗旅游需求。

验收标准：

- 能采集用户的国籍、出发地、医疗项目、出发时间、停留时长、季节灵活度和预算偏好。
- 系统推断或默认的字段应展示给用户确认。
- 对低置信字段或不确定选项应主动要求用户补充。
- 首屏不得以开放式长文本框作为主输入方式。
- 不应在用户未确认前直接进入报告生成。

### FR-002 用户画像结构化

系统应将用户信息结构化为可被多 Agent 使用的统一档案。

初版方案最低必填字段：

- 国籍
- 当前居住地
- 出发城市
- 医疗项目
- 期望出发时间

完整方案需确认字段：

- 预算等级
- 同行人数
- 是否持有中国内地身份证
- 酒店偏好
- 旅游强度
- 签证 / 入境状态

选填字段：

- 护照类型
- 语言偏好
- 是否需要接送机
- 是否需要人工翻译
- 既往相关医疗情况摘要

验收标准：

- 缺失初版方案最低必填字段时不能生成报告。
- 缺失完整方案需确认字段时，可生成暂定方案，但必须标记默认值或待确认状态。
- 敏感医疗信息必须明确标注用途。
- 用户可随时修改画像并重新生成方案。

### FR-003 医疗项目规则库

系统应维护不同医疗项目的周期、费用、限制和复查规则。

示例规则：

| 项目 | 最短周期 | 关键节点 | 风险提示 |
| --- | --- | --- | --- |
| 全飞秒近视手术 | 3-4 天 | 术前检查、手术、术后 1 天复查 | 是否适合手术需以检查为准 |
| 牙科种植 | 1 次或多次赴华 | 检查、种植、恢复、修复 | 可能需要多阶段治疗 |
| 高端体检 | 1-2 天 | 采血、影像、专科检查、报告解读 | 部分项目需空腹 |
| 医美项目 | 1-7 天 | 面诊、治疗、恢复观察 | 恢复期与项目类型相关 |

验收标准：

- 每个推荐方案必须明确最短停留天数。
- 医疗节点不能被旅游活动覆盖。
- 对医学不确定性必须展示免责声明。

### FR-004 医院 / 机构推荐

系统应根据项目、城市、预算和用户身份推荐合适的医院或机构。

推荐维度：

- 项目匹配度
- 国际部 / 特需 / 私立服务能力
- 外籍用户接待能力
- 价格区间
- 预约难度
- 交通便利性
- 语言服务
- 数据新鲜度

验收标准：

- 至少输出 1 个主推荐和最多 3 个备选推荐。
- 每个推荐必须包含推荐原因和不适合人群。
- 不得承诺实际号源可用，除非已接入真实预约系统。

### FR-005 入境与证件规则提示

系统应根据用户国籍、证件类型、停留时间和目的地提示入境要求。

验收标准：

- 签证 / 免签政策必须通过动态数据源或人工维护库获取。
- 报告中必须展示政策更新时间。
- 当政策数据不可用或置信度低时，系统应提示用户咨询官方渠道或顾问。
- 不得将过期政策作为确定结论输出。

### FR-006 航班推荐

系统应基于医疗时间窗推荐往返航班。

验收标准：

- 航班抵达时间必须满足医疗节点前置缓冲。
- 回程航班必须晚于必要复查节点。
- 展示价格、航司、中转次数、总时长和行李限制。
- 当实时价格不可用时，展示估算价和数据状态。

### FR-007 酒店推荐

系统应推荐医院周边适合外籍用户入住的酒店。

验收标准：

- 优先推荐步行 15 分钟以内或车程 20 分钟以内的酒店。
- 必须标记是否支持外宾入住；无法确认时标记为“需确认”。
- 展示价格、评分、距离、取消政策和交通方式。
- 酒店推荐不得与医疗时间表冲突。

### FR-008 本地交通规划

系统应规划机场、酒店、医院和旅游点之间的交通。

验收标准：

- 每段交通展示方式、预计时长和费用。
- 医疗节点前必须预留交通缓冲。
- 对夜间抵达、老人同行、术后行动不便场景应推荐更保守方案。

### FR-009 旅游活动推荐

系统应在医疗空隙中推荐轻度旅游活动。

验收标准：

- 活动强度必须匹配医疗项目恢复要求。
- 术后恢复期不得推荐剧烈活动、高风险活动或明显冲突活动。
- 活动应展示距离、预计时长、门票或消费预算。
- 用户可关闭旅游推荐，仅保留医疗行程。

### FR-010 全额费用测算

系统应生成完整预算明细和本国对比。

费用类别：

- 医疗费用
- 挂号 / 面诊 / 检查费用
- 药费和基础耗材
- 机票
- 酒店
- 本地交通
- 餐饮
- 签证 / 入境相关费用
- 保险
- 回国复查
- 缓冲金

验收标准：

- 同时展示人民币和用户本币。
- 汇率必须展示更新时间。
- 对费用区间使用最低、最高和推荐估算值。
- 输出净节省金额和节省比例。
- 当数据不完整时必须标记估算项。

### FR-011 报告生成

系统应生成结构化医疗旅游报告。

报告模块：

- 执行摘要
- 用户需求摘要
- 多城市方案总览
- 推荐医疗方案列表
- 每个城市方案的差旅方案
- 每个城市方案的费用 Dashboard
- 每个城市方案的小时级时间表
- 出发前准备清单
- 风险与注意事项
- 下一步行动

验收标准：

- 报告应在生成后可保存、分享和导出 PDF。
- 报告中的关键推荐应可追溯到数据源或规则。
- 报告应支持用户修改条件后重新生成。
- 初版报告默认应生成最多 4 个城市方案；每个方案必须包含时间表和费用估算。
- 用户选择主方案前，系统不得只展示单一城市作为唯一答案，除非可行城市不足 2 个并说明原因。

### FR-012 方案对比

系统应允许用户对比不同城市、医院或预算等级。

验收标准：

- 默认支持最多 4 个城市方案并排或卡片式对比。
- 每个城市方案必须包含医疗方案、差旅方案、费用估算、小时级时间表、风险提示和出发前准备影响。
- 对比维度包括总价、节省金额、天数、医疗便利性、航班便利性、酒店便利性、旅游体验、签证 / 支付准备风险和总体置信度。
- 系统必须标记推荐排序，例如“综合推荐”“最低总价”“最短行程”“医疗资源最强”。
- 用户可将其中一个方案设为主方案。
- 用户选择主方案后，系统进入该方案的详细编辑、时间表重生成和顾问转接流程。

### FR-013 人工顾问后台

系统应支持顾问查看用户授权后的需求和 AI 报告。

验收标准：

- 顾问不可查看用户未授权的敏感字段。
- 顾问可以添加备注、修改预约状态和发送补充建议。
- 所有顾问操作需要记录操作日志。

### FR-014 签证与入境准备指导

系统应根据用户国籍、护照类型、出发国家、旅行目的、停留天数和入境口岸生成签证 / 免签准备指导。

功能要求：

- 自动判断用户是否可能适用免签、过境免签或必须提前申请签证。
- 对新加坡、美国、马来西亚等首批市场展示国家 / 地区专属入口和官方链接。
- 对用户行程生成签证材料清单、办理时限、风险提示和待确认事项。
- 支持用户标记签证状态：未确认、可能免签、准备申请、已提交、已获签、需顾问确认。
- 当签证状态未确认时，报告应提示用户谨慎购买不可退机票、酒店和医疗预约服务。

签证申请步骤模板：

1. 确认是否适用免签或过境免签。
2. 若不适用，确认签证类型和申请地。
3. 检查护照有效期、空白页、照片和居住证明要求。
4. 在线填写签证申请表。
5. 准备机票、酒店、医疗预约、邀请函或其他官方要求材料。
6. 按当地签证中心 / 使领馆要求提交材料。
7. 按要求完成指纹采集、缴费、补件和护照领取。
8. 将签证结果回填到 MedTour AI，重新确认医疗和差旅方案。

验收标准：

- 签证 / 免签结论必须展示来源链接、政策更新时间和置信度。
- 系统不得将签证建议作为法律意见；必须提示以中国官方渠道和签证中心要求为准。
- 当政策信息缺失、过期或用户情况复杂时，必须标记“需顾问确认”。
- 出发前清单必须将签证状态设为高优先级阻塞项。

### FR-015 支付宝国际用户与在华支付准备

系统应引导海外用户在出发前完成在华支付准备，重点包括 Alipay 官方 App 下载、国际银行卡绑定、身份验证和备用支付方案。

功能要求：

- 根据用户设备系统展示 Alipay iOS / Android 官方下载入口。
- 说明 Alipay 面向外国访客的基础准备步骤。
- 引导用户准备可用于验证的手机号、护照信息和国际银行卡。
- 提示用户出发前完成账号注册、语言设置、身份验证、绑卡和支付密码设置。
- 提示用户不要在 MedTour AI 内输入银行卡号、验证码或支付密码；所有支付设置必须在 Alipay 官方 App 内完成。
- 在医院、酒店和交通节点展示对应支付方式可用性。
- 提供备用支付建议：境外银行卡、少量人民币现金、医院柜台支付方式确认、可选微信支付。

验收标准：

- 系统仅提供支付准备指导，不收集、不处理、不存储银行卡号、CVV、支付密码或短信验证码。
- 报告必须显示“支付方式需以商户、医院和 App 实际可用状态为准”。
- 若用户未完成支付准备，系统应在时间表中增加“抵达后支付设置缓冲”或建议联系顾问协助。
- 高端医疗方案必须在报告中明确医院接受的付款方式是否已确认。

### FR-016 极简引导式输入与渐进式补全

系统应将首轮输入压缩到生成初版方案所必需的信息，并通过选择题、短问题、搜索选择、日期控件、默认值和后续追问逐步补全用户画像。首屏不得使用开放式长文本输入作为主要入口。

首轮最低输入：

- 医疗项目或医疗意向
- 国籍 / 居住地
- 出发城市
- 大致出发时间
- 可接受停留时长
- 季节 / 淡旺季接受度，例如是否接受冬季或淡季出行

降低输入成本的交互方式：

- 医疗目的卡片：眼科、齿科、体检、医美、其他常见项目。
- 级联短问题：根据上一题答案展示下一题，例如选择眼科后再问 SMILE Pro / LASIK / ICL / Not sure。
- 国家和出发城市搜索选择：支持输入少量字符搜索，但不要求用户写长句。
- 日期控件：具体日期、月份、日期范围或 flexible。
- 停留时长选择：3-4 天、5-7 天、8 天以上、不确定。
- 季节灵活度选择：接受冬季 / 接受淡季 / 只接受指定月份 / 取决于价格。
- 智能默认值：预算、同行人数、酒店偏好、旅游强度可先由系统默认。
- 快速选择卡片：用 2-3 个选项替代长表单。
- 稍后补充：非阻塞信息不要求用户在生成前填写。
- 一键复用：注册用户可复用上一次的出发城市、护照国家、预算和酒店偏好。

验收标准：

- 用户完成首屏 5-6 个短问题后即可生成初版方案。
- 系统必须明确标记哪些字段为默认值、哪些字段由用户确认。
- 默认值影响费用或行程时，报告必须允许用户一键修改并重新生成。
- 表单不得一次性要求用户填写超过 7 个字段后才能看到初版结果。
- 对医疗安全、签证和高成本决策相关字段，不得用隐藏默认值替代用户确认。

### FR-017 Agent 不确定性追问与决策确认

系统应在 Agent 判断不确定、存在多个合理选择或决策影响较大时，主动向用户请求确认。

必须触发确认的场景：

- 医疗项目识别置信度低，例如用户只说“eye surgery”。
- 多个城市 / 医院方案接近，且总价、天数或医疗流程差异明显。
- 用户预算与推荐方案不匹配。
- 签证、免签或入境资格无法确认。
- 航班时间与医疗节点存在冲突。
- 酒店外宾入住资质无法确认。
- 用户选择不可退订机票 / 酒店前，签证或预约状态仍未确认。
- 医疗项目可能需要多次赴华或无法短期完成。

确认问题格式：

- 问题标题
- 简短原因
- 推荐选项
- 备选选项
- 选择影响说明
- 是否可稍后修改

验收标准：

- 阻塞型确认未完成时，不得输出“最终方案”，只能输出“暂定方案”。
- 每个确认请求最多展示 3 个主选项，避免把用户重新拉回复杂表单。
- 用户确认后，系统必须记录 decision_id、decision_reason、selected_option 和 affected_plan_sections。
- 用户修改确认选择后，系统只重新生成受影响部分。
- 所有低置信字段必须在报告中可见，不得静默进入最终推荐。

### FR-018 可编辑时间表与偏好驱动重新生成

系统应允许用户基于个人偏好修改时间表，并重新生成可执行行程。

可编辑项：

- 出发日期或日期范围
- 总停留天数
- 出发 / 回程时间偏好
- 航班偏好
- 酒店价格、星级、距离和外宾入住要求
- 旅游强度和景点类型
- 是否需要接送机、翻译、陪诊或无障碍支持
- 每日作息偏好
- 预算上限

不可随意编辑项：

- 医疗项目的必要检查、手术、治疗和复查节点。
- 医疗恢复期的最低休息要求。
- 医院或机构明确要求的签到和候诊时间。

验收标准：

- 用户修改偏好后，系统必须在 15 秒内返回新版时间表或失败原因。
- 新版时间表必须展示与上一版相比的变化摘要。
- 当用户偏好与医疗硬约束冲突时，系统必须解释冲突并给出可行替代方案。
- 用户可保存、对比和回退最近 3 个时间表版本。
- 重新生成时间表时，费用 Dashboard 和出发前准备清单必须同步更新受影响项。

### FR-019 时间表节点详细信息与成本拆分

系统生成的时间表必须包含每个步骤的可执行细节，用户无需离开报告即可理解每一步的时间、地点、费用、预订状态和注意事项。

通用字段：

- 节点类型
- 开始和结束时间
- 地点名称
- 完整地址
- 联系方式或服务入口
- 预订 / 预约状态
- 预计费用
- 费用币种
- 费用拆分
- 数据来源
- 数据更新时间
- 置信度
- 注意事项

航班节点必须包含：

- 航司名称
- 航班号
- 出发机场、航站楼、登机建议时间
- 到达机场、航站楼、入境缓冲时间
- 起飞时间、到达时间、飞行时长
- 中转机场和中转时长
- 票价、税费、行李费用、座位或服务费用
- 退改规则
- 价格更新时间

酒店节点必须包含：

- 酒店名称
- 星级或用户评分
- 完整地址
- 距离医院的步行时间、车程时间和距离
- 房型
- 入住和退房时间
- 每晚房价、税费、服务费、押金和总价
- 取消政策
- 是否确认支持外宾入住
- 数据来源和更新时间

医疗节点必须包含：

- 医院 / 机构名称
- 科室或中心
- 服务类型：国际部、特需、私立或普通门诊
- 完整地址
- 报到地点、楼层或窗口
- 预约状态
- 预计等待时间
- 费用拆分：挂号、检查、治疗 / 手术、药品、耗材、复查
- 需携带材料
- 术前 / 术后注意事项

本地交通节点必须包含：

- 出发地和目的地
- 建议交通方式
- 预计路线
- 预计时长
- 预计费用
- 高峰期或夜间风险提示
- 是否建议提前预约

验收标准：

- 每个时间表节点至少必须包含通用字段。
- 航班、酒店、医疗和本地交通节点必须包含对应类别的详细字段。
- 费用字段必须能汇总到费用 Dashboard，并保持数字一致。
- 当实时航班、酒店或价格数据不可用时，系统必须标记为估算值，并显示估算依据。
- 用户修改航班、酒店或时间偏好后，相关节点详情和费用拆分必须同步更新。

### FR-020 多城市候选方案生成与选择

系统应在初版报告中生成多个中国城市候选方案，让用户先比较再选择主方案。

功能要求：

- 默认生成最多 4 个城市方案。
- 每个城市方案必须是完整方案，而不是只有城市名称或医院推荐。
- 每个城市方案必须包含：
  - 城市和推荐标签
  - 推荐医院 / 机构
  - 医疗项目周期和必要节点
  - 详细小时级时间表
  - 航班方案
  - 酒店方案
  - 本地交通方案
  - 旅游 / 休息安排
  - 医疗费用估算
  - 差旅费用估算
  - 总费用估算
  - 本国价格对比和净节省
  - 签证 / 入境与支付准备影响
  - 主要风险和不确定项
  - 数据来源、更新时间和置信度
- 城市方案应尽量覆盖不同优化目标：
  - 综合推荐
  - 最低总价
  - 最短行程
  - 医疗资源最强
  - 航班最便利
- 若候选城市不足 4 个，系统必须说明原因，例如“符合该医疗项目和外籍接待能力的城市不足”。
- 用户可选择一个城市方案作为主方案，并继续编辑时间表或联系顾问。
- 用户可在已生成的城市方案之间切换，切换时保留对应方案的时间表、费用和准备清单。

验收标准：

- 初版报告至少生成 2 个可行城市方案；若少于 2 个，必须展示不可行原因和待确认项。
- 每个城市方案的费用估算必须能独立汇总，并可进入费用 Dashboard。
- 每个城市方案的时间表必须满足 FR-019 的详细节点要求。
- 多城市方案之间的比较必须展示总费用、天数、医疗置信度、航班便利性、酒店便利性、风险和推荐理由。
- 用户选择主方案后，系统应只精细化重算该方案，不丢弃其他备选方案。

## 9. Multi-Agent 需求

### 9.1 Agent 职责

| Agent | 输入 | 输出 | 关键约束 |
| --- | --- | --- | --- |
| User Profiler Agent | 引导式选择、短问题答案、表单字段、历史偏好 | 结构化用户画像、默认值、待确认字段 | 首轮输入最小化，低置信字段必须回问 |
| Medical Consultant Agent | 用户画像、项目规则 | 多城市医疗候选方案、周期、费用 | 不输出诊断结论，默认最多生成 4 个城市候选 |
| Travel Coordinator Agent | 多城市医疗时间窗、偏好 | 每个城市的航班、酒店、交通、旅游 | 医疗节点为硬约束 |
| Finance Estimator Agent | 多城市医疗与差旅数据 | 每个城市的预算、ROI 对比 | 标记汇率和数据时间 |
| Timeline Generator | 多城市医疗、航班、酒店、交通、用户偏好 | 每个城市的小时级行程、版本差异摘要 | 自动插入缓冲时间，支持偏好驱动重算 |
| Pre-Trip Readiness Agent | 用户画像、目的地、行程、政策链接 | 签证 / 支付 / 文件准备清单 | 仅输出准备指导，低置信时转人工确认 |
| Orchestrator Agent | 所有 Agent 上下文 | 最终报告 | 解决冲突，控制延迟 |

### 9.2 编排规则

- User Profiler Agent 必须先完成结构化画像。
- User Profiler Agent 应区分“用户已确认字段”“系统默认字段”“待确认字段”。
- Medical Consultant Agent 必须先输出多个城市候选及各自医疗硬性时间窗。
- Travel Coordinator Agent 必须为每个候选城市生成遵守医疗硬性时间窗的差旅方案。
- Finance Estimator Agent 必须为每个候选城市生成独立费用估算，并可与 Travel Coordinator Agent 并行执行部分估算。
- Timeline Generator 必须为每个候选城市生成基础小时级时间表；用户选择主方案后再进行精细化重算。
- Pre-Trip Readiness Agent 可在医疗城市和行程天数确定后并行生成出发前准备清单。
- 任何 Agent 触发阻塞型 confirmation_request 时，Orchestrator Agent 必须暂停最终方案锁定。
- 用户修改时间表偏好时，Orchestrator Agent 应只重跑受影响的 Agent，并保留未受影响的医疗硬约束。
- Orchestrator Agent 必须汇总多城市方案，输出推荐排序和差异摘要。
- Orchestrator Agent 必须检测冲突：
  - 航班抵达晚于检查时间。
  - 酒店距离过远导致医疗节点风险。
  - 旅游活动与术后恢复限制冲突。
  - 总预算超过用户预算等级明显上限。
  - 签证状态或支付准备未完成但用户选择不可退订方案。

### 9.3 Agent 输出质量要求

- 所有关键结论包含置信度：高 / 中 / 低。
- 所有动态数据包含来源或更新时间。
- 低置信度结论必须转化为待确认事项。
- 不允许输出“保证成功”“保证预约”“保证治疗效果”等表述。
- Agent 输出必须标记字段状态：user_confirmed、system_default、inferred、needs_confirmation。
- 多选项决策必须给出推荐选项和推荐原因，不得让用户面对无解释的列表。
- Agent 在生成追问时应优先使用选择卡片、开关或短输入，避免长问卷。

### 9.4 OpenAI API 模型调用要求

系统的核心推理 LLM 与文本向量 Embedding 能力必须通过 OpenAI API 调用实现。GCP 继续承担应用部署、Agent 编排、数据库、密钥管理、日志、监控和网络安全能力。

#### 9.4.1 推理 LLM

适用范围：

- 引导式输入理解、字段归一化与偏好摘要。
- 多语言输入翻译、归一化与摘要。
- 医疗方案、差旅方案、费用报告和时间表的结构化生成。
- Orchestrator Agent 的冲突识别、方案合并和最终报告撰写。
- 顾问后台中的 AI 摘要和下一步建议生成。

验收标准：

- 所有 Agent 的 LLM 推理调用必须走统一的 OpenAI API Client，不允许在业务代码中分散直连模型接口。
- LLM 输出必须优先要求 JSON Schema 或等价结构化格式，失败时自动重试并记录失败原因。
- 每次模型调用必须记录 trace_id、agent_name、model、latency、token_usage、status 和 error_code。
- 不得把 LLM 输出直接作为医疗诊断或正式报价展示；必须经过规则校验、免责声明注入和置信度标记。
- 模型名称应通过环境变量或配置中心管理，便于在不同推理模型之间升级或回滚。

#### 9.4.2 文本 Embedding 与 RAG

适用范围：

- 医院国际部 / 特需门诊知识库检索。
- 医疗项目周期、费用规则和注意事项检索。
- 入境政策、预约流程、外宾入住规则等运营知识检索。
- 用户历史方案、顾问备注和 FAQ 的语义检索。

验收标准：

- 所有文本向量必须通过 OpenAI text embedding model 生成。
- 文档入库时应保存 embedding_model、embedding_version、source_url、source_updated_at、ingested_at 和 chunk_id。
- 知识库检索必须先召回相关文本片段，再交由 LLM 生成答案；不得仅依赖模型记忆回答医院政策、价格、签证或流程问题。
- 当检索召回结果不足、过期或低置信时，Agent 必须输出“需确认”或触发人工顾问确认。
- 更新医院价格、签证政策或流程文档后，系统必须支持重新生成向量并保留版本记录。

#### 9.4.3 模型安全与隐私

验收标准：

- OpenAI API Key 必须存储在 GCP Secret Manager 或等价密钥管理系统中，不得写入代码仓库、前端代码或日志。
- 发送到 OpenAI API 的请求应做最小化处理，仅包含完成当前任务必要的字段。
- 护照号、联系方式、完整出生日期等强敏感字段默认不得进入模型上下文，除非用户明确授权且业务必要。
- 模型请求和响应日志应默认脱敏，不记录完整医疗自由文本或可识别个人身份的信息。
- 用户删除个人数据时，与该用户关联的模型上下文缓存、报告草稿和向量记录必须同步删除或匿名化。

#### 9.4.4 模型降级与成本控制

验收标准：

- OpenAI API 调用失败时，系统应支持有限重试、超时控制和降级提示。
- 对低风险任务可使用较低成本模型；对最终报告合成、冲突判断和医疗风险提示应使用推理能力更强的模型。
- 单次报告生成必须设置最大 token、最大重试次数和整体预算阈值。
- 当模型服务不可用导致无法生成可信报告时，系统不得输出半成品医疗方案，应提示用户稍后重试或转人工顾问。

## 11. 非功能需求

### NFR-001 性能

- 完整报告生成 P95 ≤ 15 秒。
- 页面首次可交互时间 ≤ 3 秒。
- 表单字段自动识别反馈 ≤ 2 秒。

### NFR-002 可用性

- 核心流程在移动端可完整完成。
- 用户无需注册即可生成首份基础报告。
- 表单支持保存草稿。
- 错误信息必须解释下一步，而不是只显示失败。

### NFR-003 安全与隐私

- 所有传输必须使用 HTTPS。
- 敏感字段加密存储。
- GCP 上启用 Cloud KMS、Secret Manager 和最小权限 IAM。
- 对医疗意向、国籍、联系方式等字段设置访问控制。
- 支持用户删除个人数据。
- 记录用户授权顾问查看数据的时间和范围。

### NFR-004 合规

- 遵守 GDPR 及目标市场适用的数据保护要求。
- 报告必须展示医疗免责声明。
- 签证 / 入境信息必须注明“以官方渠道为准”。
- 不将 AI 输出包装成医生诊断或正式报价。
- 签证指导不得包装成法律意见，支付指导不得要求用户在 MedTour AI 内提交银行卡敏感信息。

### NFR-005 可观测性

- 记录每个 Agent 的调用耗时、成功率、失败原因。
- 记录报告生成链路 trace_id。
- 对动态数据源失败设置告警。
- 对低置信报告比例设置监控。

### NFR-006 可靠性

- 外部 API 失败时应降级为估算模式。
- 降级结果必须明确标记“非实时数据”。
- Agent 输出解析失败时应自动重试。
- 系统应保留最近一次成功报告。

## 13. 关键边界与异常处理

| 场景 | 系统处理 |
| --- | --- |
| 用户国籍对应入境政策不可确认 | 标记为待确认，不输出确定结论 |
| 航班 API 不可用 | 使用历史估算价，明确标记非实时 |
| 医院费用缺失 | 输出费用区间，并提示需顾问确认 |
| 用户预算明显不足 | 推荐低成本城市 / 项目替代方案，或提示不建议出行 |
| 医疗项目不适合短期旅行 | 提示需要多阶段赴华或本地完成更合适 |
| 用户输入可能涉及急症 | 提示立即联系当地医疗机构，不生成旅游方案 |
| AI 方案冲突 | Orchestrator 自动重算或提示无法生成可行方案 |
| 签证状态未确认且出发日期临近 | 标记高风险，建议推迟行程或转人工顾问 |
| 官方签证政策链接不可用 | 标记为需人工确认，不输出确定入境结论 |
| 用户未完成支付宝或支付准备 | 提示准备备用支付，并在行程中加入支付确认缓冲 |
| 医院支付方式不可确认 | 报告中标记为待确认，顾问需在预约前核实 |

## 14. 示例用户旅程：新加坡用户赴上海全飞秒

### 14.1 用户输入

Sun，31 岁，新加坡籍，从新加坡出发，希望 8 月去上海做 SMILE Pro 全飞秒，预算高端，希望顺便玩 1 天。

### 14.2 系统识别

- 国籍：新加坡
- 出发城市：新加坡
- 目标城市：上海
- 医疗项目：SMILE Pro 全飞秒
- 预算等级：高端
- 旅行偏好：轻度城市游
- 医疗周期：至少 3-4 天，需以术前检查为准

- [Alipay iOS App Store](https://apps.apple.com/us/app/alipay-simplify-your-life/id333206289)：iPhone 用户下载 Alipay 官方 App。
- [Alipay Google Play](https://play.google.com/store/apps/details?id=com.eg.android.AlipayGphone)：Android 用户下载 Alipay 官方 App。

链接使用规则：

- 不同国家用户优先展示当地中国使领馆或签证中心页面。
- 若官方链接无法访问，系统应展示“需用户自行核对官方渠道”并转人工顾问。
- 不得将第三方博客作为签证结论来源；第三方文章只能作为补充阅读。
- 支付 App 下载必须优先指向官方应用商店或官方开发者页面。

## 16. 风险与合规提示

- AI 生成内容仅用于行程和预算规划，不构成医疗诊断、治疗建议或正式报价。
- 医疗项目是否可执行，应以医生面诊和检查结果为准。
- 医院价格、预约流程、入境政策、航班和酒店价格可能实时变化。
- 用户应在付款或出行前通过官方渠道或顾问再次确认。
- 系统不得诱导用户隐瞒病史或绕过医疗机构正式流程。
- 签证和入境信息仅作为准备清单，最终以中国官方渠道、签证中心和边检机关解释为准。
- 支付宝、银行卡和移动支付设置必须在官方 App 或银行渠道完成，MedTour AI 不接收用户支付凭证。

## 17. 里程碑

| 阶段 | 目标 | 交付物 |
| --- | --- | --- |
| M0 原型 | 验证核心报告体验 | 表单、静态报告、费用样例、时间表样例 |

| M3 顾问闭环 | 支持线索转接和顾问后台 | CRM / 后台状态流转 |
| M4 商业化 | 支持合作机构、转化追踪和多市场扩展 | 推荐策略、运营后台、数据报表 |

## 18. 待确认问题

- 首批上线国家是否限定为新加坡、美国、马来西亚？
- 首批医疗项目是否聚焦眼科和齿科，还是同时覆盖体检与医美？
- 医院数据来源是公开资料、合作机构，还是人工维护知识库？
- 是否需要接入真实预约系统，还是 MVP 仅做顾问转接？
- 报告导出是否需要品牌化 PDF 模板？
- 是否允许用户上传检查报告或处方等更敏感医疗文件？
- 顾问后台是否接入现有 CRM？
- 是否需要为签证申请提供人工代办服务，还是仅提供官方链接和清单？
- 是否需要在报告中加入 Alipay 之外的 WeChat Pay 设置指南？
- 是否需要在顾问后台记录用户签证和支付准备完成度？
